# PRD — Local LLM Bench (מטלה 5: AirLLM מול הרצה סטנדרטית)

**קוד קבוצה:** uoh-rl07
**גרסה:** 1.00
**סטטוס:** בתכנון/פיתוח שלד — הרצת ניסויים בפועל טרם בוצעה.

## 1. סקירת הפרויקט והקשר

מטלה 5 בקורס "סוכני AI" (תשפ"ו, סמסטר ב) עוסקת בהרצה מקומית של מודל שפה גדול תוך
השוואה מעמיקה בין הרצה סטנדרטית (טעינת המודל במלואו לזיכרון) לבין הרצה באמצעות
[AirLLM](https://github.com/lyogavin/airllm) (טעינה שכבה-אחר-שכבה, מבוססת `mmap`),
כולל הדגמת השפעת קוונטיזציה. הפרויקט בנוי כתוכנה מקצועית לפי
`software_submission_guidelines-V3.pdf` — לא סקריפט חד-פעמי אלא חבילת Python
מודולרית, עם SDK יחיד, שכבת גייטקיפר ל-API-ים חיצוניים, בדיקות, ותיעוד מלא.

### מפרט החומרה של סביבת הפיתוח/הרצה (רלוונטי ישירות לתכנון הניסוי)

| רכיב | ערך |
|---|---|
| CPU | Intel Core i7-1165G7, 4 cores / 8 threads |
| RAM | ~76 GB |
| GPU | Intel Iris Xe (משולב, **ללא CUDA** — אין GPU ייעודי) |
| דיסק פנוי | ~172 GB |

היעדר GPU ייעודי הופך את התרחיש הזה לדוגמה טבעית וישירה בדיוק לתוכן פרק 8 בחוברת
ההרצאה ("AirLLM: הרצת מודלים ענקיים על CPU") — כל ההשוואה מתבצעת על CPU בלבד, כאשר
ה"יתרון" של AirLLM אינו VRAM מוגבל (אין GPU בכלל) אלא ניהול חכם יותר של ה-RAM דרך
עקרונות מערכת ההפעלה (Virtual Memory, mmap, Page Cache) שנלמדו בהרצאה.

## 2. בעיית המשתמש (Problem Statement)

תלמיד/ת מדעי המחשב שרוצה להריץ מודל שפה "גדול" (במונחי הזמינות הציבורית ב-Hugging
Face) על מחשב אישי ללא GPU נתקל/ת בבעיה כפולה: (א) טעינת המודל המלא לזיכרון עלולה
לצרוך כמעט את כל ה-RAM ולגרום להאטה דרסטית או לכשל; (ב) אין כלי סטנדרטי שמראה
בבירור *כמה* משתלם להשתמש בטכניקות כמו AirLLM וקוונטיזציה, ובאילו תנאים. הפרויקט
בונה כלי מדידה (benchmark) אובייקטיבי שמריץ את אותו המודל בשלוש שיטות ומפיק דוח
משווה מבוסס-נתונים.

## 3. קהל יעד

- תלמידה (בוני הפרויקט) — לצורך הגשת המטלה.
- בודק/ת המטלה (מרצה הקורס / סוכן AI בודק) — צריך/ה להיות מסוגל/ת להבין את הממצאים
  במהירות מהדוח, מהגרפים, ומה-README.
- כל מפתח/ת עתידי/ת שירצה להריץ benchmark דומה על מודל/חומרה אחרים (הרחבה).

## 4. יעדים ומדדי הצלחה (KPIs / Acceptance Criteria)

| # | יעד | קריטריון קבלה |
|---|---|---|
| G1 | הרצת Baseline מוצלחת/כושלת מתועדת | נמדדו ותועדו: זמן טעינה, peak RAM, tokens/sec, זמן כולל |
| G2 | הרצת AirLLM על אותו מודל | נמדדו אותם מדדים; הפרש peak RAM מול Baseline ≥ פי 3 (צפי) |
| G3 | הדגמת קוונטיזציה | לפחות 2 רמות קוונטיזציה (למשל Q4_K_M, Q2_K) נבדקו מול ה-FP16/FP32 baseline |
| G4 | ניתוח רגישות פרמטרים | סריקת אורך פרומפט/מספר טוקנים עם גרף תוצאה לכל שיטה |
| G5 | עמידה בסטנדרט התוכנה | כיסוי בדיקות ≥85%, `ruff check` נקי, כל התיעוד החובה קיים |
| G6 | שחזוריות | כל הרצה מלאה אפשרית דרך `uv run` בלבד, ללא ערכים מוקשחים |

## 5. User Stories

- **כתלמידה**, אני רוצה להריץ פקודה יחידה (`uv run python -m local_llm_bench.main
  --mode baseline`) ולקבל מדדי ביצועים, כדי שאוכל להשוות בקלות בין השיטות.
- **כתלמידה**, אני רוצה שכל קריאה חיצונית (הורדת מודל, קריאה ל-Ollama) תעבור דרך
  gatekeeper עם rate limiting, כדי שלא אחרוג בטעות ממכסות/אעמיס על השירות.
- **כבודק/ת המטלה**, אני רוצה notebook עם גרפים מוכנים, כדי להבין את הממצאים מהר.
- **כמפתחת עתידית**, אני רוצה להחליף מודל/precision רק דרך קובץ קונפיגורציה, בלי
  לגעת בקוד.

## 6. דרישות פונקציונליות

1. טעינה והרצה של מודל HF סטנדרטי (Baseline) עם מדידת latency/memory.
2. טעינה והרצה של אותו מודל דרך AirLLM עם אותם מדדים.
3. הרצת המודל בקוונטיזציה (דרך Ollama/GGUF) באותם מדדים + השוואת איכות פלט איכותנית.
4. ריצת benchmark מלאה (כל 3 השיטות, מספר פרומפטים, מספר אורכי פלט) ואגירת התוצאות
   כ-JSON/CSV תחת `results/`.
5. הפקת גרפים/טבלאות מתוך תוצאות שמורות (ללא תלות בהרצה מחדש).
6. זיהוי אוטומטי של מפרט החומרה (CPU/RAM/GPU) לשילוב בדוח.

## 7. דרישות לא-פונקציונליות

- כל תצורה (מודל, precision, מספר טוקנים, timeouts, rate limits) דרך
  `config/setup.json` / `config/rate_limits.json` / `.env` — איסור מוחלט על ערכים
  מוקשחים בקוד.
- קבצי קוד ≤150 שורות; ארכיטקטורת SDK יחיד; `ApiGatekeeper` מרכזי לכל קריאה חיצונית.
- כיסוי בדיקות ≥85% תוך שימוש ב-mocks לקריאות הכבדות (הורדת מודלים, הרצת Ollama),
  כך שהריצה הרגילה של `pytest` לא תדרוש רשת/הורדות.
- ניהול גרסאות גלובלי, החל מ-1.00.
- אבטחת מידע: HF token אך ורק דרך משתנה סביבה (`.env`, לא ב-git).

## 8. הנחות, תלויות, מגבלות ומחוץ לתחום

**הנחות:** למחשב יש חיבור אינטרנט להורדת מודלים בשלב ההרצה; מותקן Python 3.10+
ו-`uv`; Ollama מותקן מקומית לניסוי הקוונטיזציה.

**תלויות חיצוניות:** Hugging Face Hub (משקלות מודל), חבילת `airllm`, Ollama
(CLI + API מקומי על `localhost:11434`).

**מגבלות:** אין GPU — כל ההרצות הן CPU-only; ריצות על מודל ~14B עשויות לקחת זמן ניכר
(דקות עד עשרות דקות לכל ניסוי) — מתועד ומתוכנן מראש (ר' `docs/PLAN.md`, ADR-1).

**מחוץ לתחום:** פיתוח/אימון מודל חדש; ממשק גרפי (GUI); פריסה לענן/שרת; מילוי טופס
ה-Word/PDF הרשמי להגשה במודל (מטופל ידנית על ידי התלמידה בנפרד, לפי ההנחיות — אין
לגעת בשדותיו).

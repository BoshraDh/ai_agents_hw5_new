### מפרט החומרה של סביבת הפיתוח/הרצה (רלוונטי ישירות לתכנון הניסוי)

| רכיב | ערך |
|---|---|
| CPU | Intel Core i7-1165G7, 4 cores / 8 threads |
| **RAM** | **~7.65 GB** (8,211,927,040 בייטים — `Win32_OperatingSystem.TotalVisibleMemorySize`) |
| GPU | Intel Iris Xe (משולב, **ללא CUDA** — אין GPU ייעודי, VRAM לא רלוונטי) |
| אחסון | SSD |

היעדר GPU ייעודי הופך את התרחיש הזה לדוגמה טבעית וישירה בדיוק לתוכן פרק 8 בחוברת
ההרצאה ("AirLLM: הרצת מודלים ענקיים על CPU") — כל ההשוואה מתבצעת על CPU בלבד. **עם
RAM כה מצומצם (~7.65GB), כמעט כל מודל בגודל בינוני-גדול ומעלה נכשל בטעינה ישירה

אחת בזיכרון בכל רגע, בסדר גודל של מאות MB בודדות) הופך לקריטי ולא רק ל"נחמד שיהיה",
בהתבסס על עקרונות Virtual Memory, mmap, ו-Page Cache שנלמדו בהרצאה.

## 1.1 שאלות מחקר (ex05 §4) — חובה לענות עליהן בדוח הסופי

1. מהו צוואר הבקבוק האמיתי שמונע את ההרצה הישירה — זיכרון (RAM/VRAM) או כוח
   חישוב? כיצד זיהינו זאת בפועל (מדידה, לא ניחוש)?
2. כיצד AirLLM משנה את הקצאת המשאבים, ומהו הקשר לזיכרון הווירטואלי ולמנגנון
   ה-Paging של מערכת ההפעלה?
3. מה הייתה השפעת הקוונטיזציה על צריכת הזיכרון, על המהירות, ועל איכות הפלט?
   היכן עבר "קו האדום" של הדיוק?
4. כיצד באים לידי ביטוי שלבי Prefill/Decode במדידות שלנו, וכיצד הם משתקפים
   בהפרדה בין מדד TTFT (Time To First Token) לבין מדד TPOT (Time Per Output
   Token)?
5. מהו המחיר (Latency/Throughput) שאתם משלמים עבור היכולת להריץ מודל גדול על
   חומרה צנועה?
6. מתי כדאי כלכלית לעבוד מקומית, ומתי עדיף להשתמש ב-API חיצוני? (ר' סעיף 6
   למטה — ניתוח כלכלי מחייב).

## 1.2 הערה חשובה — היקף הניסוי ובחירת גודל המודל (ex05, "שיקול חשוב")

מטרת המטלה **אינה** להריץ מודלים במשך שעות/ימים. יש לשקול את כוח החישוב של המחשב
האישי, ולבחור מודל בהתאם לכך **שיהיה גדול — אך לא גדול מדי**: מודל כזה אמור להדגים
latency ארוכה או חוסר יכולת להריץ ישירות, אל מול השיפור וההקלה בעזרת AirLLM.
**זכרו: ניסוי שלילי הוא גם תוצאה רצויה** — אם לא הצלחתם לשפר את הביצועים, מותר
ואף רצוי להציג תוצאה שלילית ולנמק ולהסביר מדוע הדברים התנהגו כך במקרה שלכם.
תוצאה שלילית שנותחה היטב שווה לא פחות מתוצאה חיובית מוצגת חובבנית.

**מכאן נגזרת דרישה נוספת שלא הייתה מתועדת קודם**: אסור לבחור מודל ענק שאין שום
סיכוי להריץ אפילו עם AirLLM (ex05 §6.2, "Don't"). המודל שנבחר (`Phi-3-medium-4k-
instruct`, ~14B) עומד בשני התנאים — **אך המשמעות השתנתה לאחר תיקון מדידת ה-RAM**
(היה ~76GB, בפועל ~7.65GB, ר' §1 למעלה): ב-FP32 (~28GB) או אפילו FP16 (~14GB) המודל
פשוט **לא ייכנס בכלל** ל-7.65GB RAM — כלומר Baseline צפוי להיכשל **מיד** בטעינה
(לא "ריצה איטית" אלא כישלון טעינה מובהק), בדיוק כמו שכבר נצפה בפועל עם
`qwen2.5:72b` דרך Ollama (ר' `README.md` יומן ניסויים, ניסוי 2). AirLLM עדיין
צפוי להצליח כי הוא לעולם לא מחזיק יותר משכבה אחת (מאות MB) בזיכרון בו-זמנית —
כך שהניגוד בין הכישלון להצלחה יהיה **חד וברור עוד יותר** מהתכנון המקורי.

## 2. בעיית המשתמש (Problem Statement)

תלמיד/ת מדעי המחשב שרוצה להריץ מודל שפה "גדול" (במונחי הזמינות הציבורית ב-Hugging
Face) על מחשב אישי ללא GPU נתקל/ת בבעיה כפולה: (א) טעינת המודל המלא לזיכרון עלולה
לצרוך כמעט את כל ה-RAM ולגרום להאטה דרסטית או לכשל; (ב) אין כלי סטנדרטי שמראה
בבירור *כמה* משתלם להשתמש בטכניקות כמו AirLLM וקוונטיזציה, ובאילו תנאים. הפרויקט
בונה כלי מדידה (benchmark) אובייקטיבי שמריץ את אותו המודל בשלוש שיטות ומפיק דוח
משווה מבוסס-נתונים.

## 5. User Stories

- **כתלמידה**, אני רוצה להריץ פקודה יחידה (`uv run python -m local_llm_bench.main
  --mode baseline`) ולקבל מדדי ביצועים, כדי שאוכל להשוות בקלות בין השיטות.
- **כתלמידה**, אני רוצה שכל קריאה חיצונית (הורדת מודל, קריאה ל-Ollama) תעבור דרך

- **כבודק/ת המטלה**, אני רוצה notebook עם גרפים מוכנים, כדי להבין את הממצאים מהר.
- **כמפתחת עתידית**, אני רוצה להחליף מודל/precision רק דרך קובץ קונפיגורציה, בלי
  לגעת בקוד.

## 6. דרישות פונקציונליות

1. טעינה והרצה של מודל HF סטנדרטי (Baseline) עם מדידת TTFT, TPOT, throughput,
   peak RAM, זמן כולל, וצריכת חשמל **משוערת** (ex05 §5.4).
2. טעינה והרצה של אותו מודל דרך AirLLM עם אותם מדדים בדיוק (השוואה ישירה).
3. הרצת המודל בקוונטיזציה (דרך Ollama/GGUF, לפחות 2 רמות) באותם מדדים + השוואת
   איכות פלט איכותנית לכל רמת קוונטיזציה.
4. ריצת benchmark מלאה (כל 3 השיטות, מספר פרומפטים, מספר אורכי פלט — ניתוח
   רגישות OAT) ואגירת התוצאות כ-JSON/CSV תחת `results/`.
5. **ניתוח כלכלי (חובה)**: חישוב עלות API חיצוני (per-token, לפי ספק לבחירה),
   חישוב עלות On-Prem (CAPEX מופחת + OPEX חשמל/תחזוקה), מציאת נקודת האיזון
   (break-even) כפונקציה של נפח שימוש, והפקת גרף עלות מצטברת. אופציונלי: השוואה
   שלישית מול GPU בענן. יש לקחת בחשבון גם Prompt/Context Caching כתופעה שעשויה
   להזיז את נקודת האיזון.
6. הפקת גרפים/טבלאות מתוך תוצאות שמורות (ללא תלות בהרצה מחדש), כולל Model
   Roofline diagram (ההרחבה המקורית שנבחרה).
7. זיהוי אוטומטי של מפרט החומרה (CPU/RAM/GPU/אחסון) לשילוב בדוח.
8. קישור מפורש בין כל ממצא למושגי ההרצאה (Prefill/Decode, VRAM, זיכרון וירטואלי,
   memory-bound מול compute-bound) — ר' `docs/PLAN.md`.

## 8. הנחות, תלויות, מגבלות ומחוץ לתחום

**הנחות:** למחשב יש חיבור אינטרנט להורדת מודלים בשלב ההרצה; מותקן Python 3.10+
ו-`uv`; Ollama מותקן מקומית לניסוי הקוונטיזציה.

**תלויות חיצוניות:** Hugging Face Hub (משקלות מודל), חבילת `airllm`, Ollama
(CLI + API מקומי על `localhost:11434`).

**מגבלות:** אין GPU — כל ההרצות הן CPU-only; ריצות על מודל ~14B עשויות לקחת זמן ניכר
(דקות עד עשרות דקות לכל ניסוי, ואף שעות ל-baseline ב-FP32 שעלול "לתקוע" את המערכת
לפי אזהרת `ex05` §11.2) — מתועד ומתוכנן מראש (ר' `docs/PLAN.md`, ADR-1). קבצי
ה-shards של AirLLM (SafeTensors רבים) חייבים להישמר בנתיב מוגדר במפורש
(`config/setup.json: airllm.layer_shards_saving_path`) כדי לא למלא את כונן ה-OS
במהלך הניסוי (ex05 §6.1, "Do").

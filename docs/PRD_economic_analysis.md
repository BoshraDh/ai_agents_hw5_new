# PRD — מנגנון: Economic Analysis (On-Prem מול API)

**מקור הדרישה:** `ex05-AirLLM.pdf.pdf` §5.5 — "זוהי דרישה כללית ומחייבת ולא
אופציונלית". סעיף זה מרחיב את `docs/PRD_benchmark_reporting.md` עם המנגנון
הכלכלי הספציפי.

## תיאור מפורט

`CostAnalysisService` מבצע חישוב עלויות ומשווה בין שתי דרכים (ואופציונלית שלישית)
לביצוע **אותה משימה בדיוק**, כדי להראות מתי כדאי כלכלית לעבוד מקומית (On-Prem)
לעומת שימוש ב-API חיצוני של ספק צד-שלישי:

1. **עלות API של צד-שלישי**: חישוב מספר הטוקנים (קלט + פלט) לכל בקשה, הכפלה
   במחיר לטוקן של ספק לבחירה (למשל OpenAI, Claude). הצגת עלות לבקשה בודדת ועלות
   לכמות בקשות נתונה.
2. **עלות עבודה מקומית (On-Prem)**: חישוב עלות החומרה (CAPEX) מופחתת על פני זמן,
   בתוספת עלות חשמל והתחזוקה (OPEX). גזירת ממנה עלות אפקטיבית לבקשה כפונקציה של
   נפח השימוש.
3. **(אופציונלי) GPU בענן**: עלות ביצוע אותה משימה על כרטיס GPU מושכר אצל ספק
   ענן (Cloud GPU) — למשל מחיר לשעת GPU כפול זמן הריצה.

על בסיס שני/שלושת החישובים, מוצאת **נקודת האיזון** (break-even): מאיזה נפח שימוש
(מספר בקשות או טוקנים בחודש) הפכת העבודה המקומית לזולה יותר מ-API. מוצג כגרף של
עלות מצטברת מול נפח שימוש, עם המלצה מנומקת: באילו תרחישים API עדיף, ובאילו
On-Prem עדיף (כולל שיקולי פרטיות/אבטחת מידע, לא רק עלות).

### רקע — תופעת Prompt/Context Caching

ספקים מבוססי ארכיטקטורת PagedAttention שומרים בזיכרון חלקים קבועים של הפרומפט
המערכתי חוזר, ללא חישוב Prefill חוזר, ומציעים תעריף מוזל משמעותית לאותם טוקנים
מוכמנים. בתרחישים של שאלות חוזרות על מסמך ארוך, מנגנון זה עשוי להזיז את נקודת
האיזון. `CostAnalysisService` חושף פרמטר `cached_token_discount_ratio`
(מקונפיגורציה) שמאפשר לשקלל זאת בחישוב עלות ה-API.

## דרישות ספציפיות

- קלט: `ApiPricingAssumptions` (מחיר לטוקן קלט/פלט, הנחת caching), `OnPremAssumptions`
  (עלות חומרה, אורך חיים בשנים, מחיר חשמל לקוט"ש, הספק ממוצע בוואט, עלות תחזוקה
  שנתית), `usage_volumes: list[int]` (נפחי שימוש חודשיים לבדיקה, למשל [10, 100,
  1_000, 10_000, 100_000] בקשות/חודש).
- כל ההנחות (מחירים, נפחי שימוש, אורך חיי חומרה, תעריף חשמל) נטענות מ-
  `config/economic_assumptions.json` — **איסור מוחלט על מחירים מוקשחים בקוד**
  (עקרון guidelines §7.2), ומתועדות בבירור ב-README כך שהניתוח שקוף וניתן לשחזור.
- פלט: `BreakevenResult` (dataclass) עם `usage_volumes`, `api_costs`,
  `onprem_costs`, `breakeven_volume: int | None` (None אם אין חיתוך בטווח שנבדק),
  ורשימת ההנחות ששימשו לחישוב (לשקיפות בדוח).
- `ReportService.plot_breakeven(result)` מפיק גרף עלות מצטברת (שני/שלושה קווים)
  מול נפח שימוש, עם סימון נקודת האיזון.

## אילוצים וחלופות שנשקלו

- נשקל לחשב עלות On-Prem פר-שעת-ריצה בלבד (ללא הפחתת CAPEX); נדחה כי זה מתעלם
  מהעלות ההתחלתית הממשית של החומרה — ה-`ex05` מבקש חישוב CAPEX+OPEX מלא.
- נשקל לקבע ספק API יחיד (OpenAI בלבד); נדחה — `config/economic_assumptions.json`
  תומך ברשימת ספקים כדי לאפשר השוואה בין כמה ספקים באותו גרף כהרחבה עתידית.
- אופציית ה-GPU בענן ממומשת כפרמטר אופציונלי (`include_cloud_gpu: bool`) ולא
  כברירת מחדל, כדי לא לסבך את הגרף הבסיסי הנדרש.

## קריטריוני הצלחה ותרחישי בדיקה

- **Happy path**: בהינתן הנחות תקינות ונפחי שימוש עולים, מתקבל `BreakevenResult`
  עם עלויות עולות מונוטונית עבור שתי השיטות, ונקודת חיתוך אחת לכל היותר (הבעיה
  ליניארית בנפח השימוש בשני הצדדים).
- **Edge case — On-Prem תמיד יקר יותר בטווח שנבדק**: `breakeven_volume=None`,
  מתועד בדוח כממצא לגיטימי (למשל אם נפחי השימוש שנבדקו קטנים מדי).
- **Edge case — הנחות חסרות/שליליות**: ולידציה מוקדמת (`ValueError`) אם מחיר
  שלילי או אורך חיים ≤0 סופק בקונפיגורציה.

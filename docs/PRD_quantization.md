# PRD — מנגנון: Quantization Service

## תיאור מפורט

`QuantizationService` מריץ את אותו מודל (או וריאנט GGUF שלו) דרך **Ollama**
המקומי, בשתי-שלוש רמות קוונטיזציה (למשל `Q4_K_M` כברירת מחדל ו-`Q2_K` כניסוי קיצון,
לפי הטבלה בהרצאה: FP32→FP16→FP8→Q4→Q2). המטרה: להדגים בפועל את אפקט הקוונטיזציה
על זיכרון, מהירות, ואיכות פלט — דרישה מפורשת במטלה ("יש להשתמש בקוונטיזייר ולהראות
את השפעתו").

### רקע תיאורטי (מההרצאה)

קוונטיזציה מקטינה את מספר הביטים לכל משקל (למשל מ-32 ל-4), מה שמקטין באופן ליניארי
כמעט את טביעת הזיכרון וגם מאיץ קריאה מדיסק/RAM (פחות בייטים לתנועה) — אך פוגעת
בדיוק המספרי ועלולה לפגוע באיכות הפלט. ההרצאה ממליצה להתחיל פיתוח עם Q2 רק כדי
לוודא שהצנרת עובדת, ולא להעריך איכות לפיו — עיקרון שמיושם כאן: קודם `Q2_K` כבדיקת
עשן טכנית, ואז `Q4_K_M` כמדד האיכות/ביצועים העיקרי.

## דרישות ספציפיות

- קלט: `model_name`, `quant_level: Literal["Q4_K_M","Q2_K"]`, `prompt`,
  `max_new_tokens`. פלט: אותו `RunMetrics`, עם שדה נוסף `quality_note: str | None`
  (הערכה איכותנית ידנית/heuristic של הפלט מול ה-baseline).
- **TTFT/TPOT (v1.01)**: בניגוד ל-`ModelLoaderService`/`AirllmService` (הזקוקים
  ל-streaming ידני, ר' `docs/PLAN.md` ADR-4), תשובת ה-API הרגילה (non-streaming)
  של Ollama **כבר כוללת** `prompt_eval_duration` (ננושניות — משמש כ-TTFT/Prefill
  proxy) ו-`eval_duration`/`eval_count` (ממוצע = TPOT/Decode proxy). השירות מחלץ
  את השדות הללו ישירות מהתשובה, ללא צורך במדידה עצמאית.
- כל תקשורת עם Ollama (`ollama pull`, קריאת API על `localhost:11434`) עוברת דרך
  `ApiGatekeeper` — כולל retry אם השירות המקומי עדיין לא עלה (`ollama serve`).
- השירות **לא** מתקין/מפעיל את Ollama בעצמו — בודק זמינות (`GET /api/tags`) ומחזיר
  שגיאה ברורה עם הוראת התקנה אם לא זמין (fail fast, לא side-effect מוסתר).

## אילוצים וחלופות שנשקלו

- נשקל להשתמש ב-`bitsandbytes`/`QLoRA` NF4 ישירות דרך `transformers` במקום Ollama;
  נדחה לטובת Ollama כי הוא מדגים גם את פורמט GGUF וה-Modelfile שנלמדו בהרצאה, וגם
  פשוט יותר להרצה מקומית ללא GPU.
- רמת `Q8`/`FP8` הושמטה מהברירת מחדל (זמן/מקום) אך מוגדרת כאופציה ב-
  `config/setup.json.quantization.available_levels` להרחבה עתידית ללא שינוי קוד.

## קריטריוני הצלחה ותרחישי בדיקה

- **Happy path**: `ollama pull` מצליח (מדומה ב-mock ליחידה), generate מחזיר טקסט,
  `peak_ram_mb` נמוך משמעותית מ-Baseline (FP32).
- **Edge case — Ollama לא רץ**: קריאת health-check נכשלת → שגיאה ברורה, לא תקיעה.
- **Edge case — רמת קוונטיזציה לא נתמכת**: ולידציה מול `available_levels` לפני
  קריאה חיצונית כלשהי.
